## #1 术语翻译腔 —— 升级中文 wrapper（已 A/B 验证）

**根因（已定位）**：SOP 的 prompt 正文是**英文指令**（`buildHumanSopPrompt` 等含 Signature Move / cognitive lever / Pattern Break / Reframe / CTA 等概念词），中文输出时仅用 `clerk.ts:20` 的 `CHINESE_WRAPPER`（内容只有一句"用简体中文写"）包一层。模型于是**现场直译**英文概念，生造出"签名式动作 / 社会仪式 CTA / 认知基模炸弹 / 主题聚类"。

**生产实证**：39 篇 zh SOP 共 17 处生造词；其中一篇 human SOP 同时命中 `社会仪式×3 / 签名式×3 / 认知基模×1 / 主题聚类×3 / 炸弹×1`。问题真实、可复现。

**A/B 实测（同输入，只换 wrapper，DeepSeek Pro）**：
- hottest SOP：V0 开头是典型 AI 寒暄"好的，以下是根据您提供的……完整转录进行的深度结构拆解。所有内容均使用简体中文撰写……"；V1 直接进正文，并主动用上"黄金前 3 秒""打断刷视频的惯性"（正是 pattern interrupt 的指定替代）。
- human SOP：V0 用生硬直译（操作手册 / 共同主题 / 标志性动作 / 叙事弧线）；V1 用中文创作者圈通用词 —— `干货×6 / 二次抓人×4 / 常见主题×4 / 打法×2 / 留扣子×2 / 完播 / 留人 / 打断刷视频`，生造词与 AI 套话均为 **0**。

**结论 / 推荐**：不重写 prompt（大改、风险高、会丢掉调好的英文结构），只把 `CHINESE_WRAPPER` 升级为「术语表 + 去翻译腔规则」。同一个 wrapper 也被 Bible（`poet.ts:66`）复用，等于一处改、Clerk SOP 与 Bible 同时受益。

**待你确认**：① 术语表（下方）是否定稿；② 旧 SOP 不会自动变 —— 是否顺手重跑 demo 频道（红发魔女 / 相机口播）给客户看效果。

**改动**：`packages/shared/src/prompts/clerk.ts` 的 `CHINESE_WRAPPER`（动了 shared → 需 Trigger 重部署）。

**验证过的新 wrapper（可直接用）**：

## #3 Muse 导入对标账号 —— 提为一等表（长期成熟方案）

> **归属：Part 2（整体大改），Part 1 不实施。** 对标账号要建成 `competitor_accounts` 一等表，而这张表是重构方案一的地基（见 [重构设计-方案一.md](./重构设计-方案一.md) §2）—— 随重构一次性建，避免先建 channels 版再拆。下面的列设计 / URL 处理 / UX / 边界到 Part 2 仍照用。

**现状**：`channels.competitors` 是 JSONB（`{platform,url}`，无 name/id/dedup）。其实有一个**很粗糙**的编辑入口（`edit-channel-sheet.tsx:237` 的 textarea，靠 `url.includes("xiaohongshu")` 猜平台、无去重、无自我导入防护），但**在 Muse 页面里没有任何入口** —— 用户正是在 Muse 想加对标却找不到。monitor 只需要 `{platform,url}` 即可（URL 直接喂 yt-dlp / TikHub）。

**成熟方案（推荐）**：把对标账号提升为**一等表**，而不是塞在 channels 的 JSONB 里。理由：后续 Project 重构要求"在 Project 内导入对标""Clerk 全局化"，JSONB 方案无法跨 Project 共享、无法存解析元数据、并发改写易丢更新。

```
competitor_accounts                       channel_competitors（关联表）
  id / user_id / platform                   channel_id  -> channels
  platform_key  -- 规范化去重键(小写)         competitor_id -> competitor_accounts
  url / display_name / platform_channel_id   PRIMARY KEY(channel_id, competitor_id)
  meta(jsonb 粉丝数等) / created_at
  UNIQUE(user_id, platform, platform_key)  -- DB 级去重（现在缺的就是这个）
```

> **注（已采用方案一）**：上表为示意，**关联维度以 [重构设计-方案一.md](./重构设计-方案一.md) §2 为准** —— 对标账号被 **Project / 我的账号** 引用，不是 `channels`。实施时直接按重构设计建表，**不要先建 channels 版**。本节的列设计 / URL 处理 / UX / 边界仍然有效、照用。

**URL 处理复用现成函数**（已用临时脚本验证 12 个 URL 全部正确 detect/validate/dedup/normalize）：
- 平台识别：`isValidXhsProfileUrl`（`xhs.ts:83`）/ `isValidYoutubeChannelUrl`（`tikhub.ts:29`）
- 规范化键：`extractXhsUserId`（`xhs.ts:57`，**键要小写**）/ `parseYoutubeChannelUrl`（`youtube-data.ts:160`，handle 小写）
- 解析名称：`resolveXhsUser`（`xhs.ts:337`）/ `fetchChannelMetaByHandle|ById`（`youtube-data.ts:145`），best-effort，超时也先存 URL

**UX**：主入口放 **Muse 频道页**（把 `muse/[slug]/page.tsx:379` 那句"先在频道设置中添加"换成真正的「导入对标账号」按钮），让"导入 → 开始巡视"在一页完成；channels 编辑页保留同一组件做平价入口。输入支持多行粘贴（每行一个 URL，逐行 ✓/✗）。

## #4 Bible —— 多版本（纯前端）+ 生成提速（已测）

### 4a 多版本并行

**实证**：`poet_bible` 无 `channelId` 唯一约束 + 有 `isActive`，**DB 早就支持一个频道多份 Bible**；"单一激活"只由应用代码维持。页面也已经查出完整版本列表。缺口**纯在前端**：
- "新建 Bible"入口只在"无激活 Bible"时出现 → 改成**常驻「+ 新建 Bible」**；
- 生成新的会把旧的置非激活（有覆盖感）→ 新建默认**不强制顶替**，由用户再激活；
- 历史列表只显示非激活、且只能激活/删除 → 改成**统一版本库**（含激活项），命名/切换/编辑齐全。

**改动**：前端 + 复用现有 `generateBible`/`activateBible`/`updateBible` mutation。无需 schema 迁移（可选加 `(channel_id) WHERE is_active` 部分唯一索引硬化单激活不变量）。

### 4b 生成提速（实测）

**实证**：13 次真实运行 min 53s / 中位 89s / **均值 94s** / max 154s；Bible 正文 2000–4200 字（最长 ≈ 3500 output token，逼近现 4096 上限、余量小）。瓶颈 = Pro 推理模型单发 + 空输出重试翻倍（max 154s 即此）+ 首次的 derive 步。

**同 prompt 配置实测**：

| 配置 | 耗时 | 字数 | 完整 |
| --- | --- | --- | --- |
| pro_4096（现状） | 63s | 2638 | ✓ |
| pro_2800 | 52s | 2479 | ✓ |
| **flash_2800** | **26s** | 3259 | ✓ |
| flash_4096 | 18s | 2131 | ✓ |

Flash 比 Pro 快 **2.4–3.5×**，产出**完整且质量在线**（实测徕卡频道 Bible：M6/M11/Noctilux 具体型号、内容支柱、地道口吻）。

**推荐**：Bible 主生成**改用 Flash**（结构化策略文，非深推理）+ **streaming**（边出边显）。`maxOutputTokens` **调高到 16384，不要降**（现 4096 对长 Bible 偏紧、有截断风险；提速靠 Flash 不靠砍输出）。改 Flash 还顺带消除空输出重试这个延迟尖刺。预期 ~94s → ~20–26s。

**待你确认**：Flash 质量是否接受（建议先多跑 3–5 条不同频道眼检再切；我已存 1 份样本在 `/tmp/bible_flash_2800.md`）。

**改动**：`packages/shared/src/services/poet/bible.ts`（模型 + maxTokens）、`apps/jobs/trigger/generate-bible.ts`（streaming）→ 需 Trigger 重部署。

## #6 时长 ≤60s 自定义 —— 改 `durationSeconds`（整数秒）

**实证**：时长全链路已摸清。后端 `computeTargetWordCount` / `LONG_FORM_THRESHOLD`（zh 2000 / en 1500）在 <60s 下行为正确（30s→100 字，仍走短稿单发，不触发长稿机制）。**真正卡点**是两个：① tRPC 校验 `z.number().int().min(1).max(60)` 的 `.int().min(1)` 把 60s 以下挡死；② DB 列是 INTEGER 分钟，存不了 0.5 分钟。

**推荐**：把分钟改为**整数秒 `durationSeconds`**。比"分钟允许小数"更干净 —— 整数秒完美对应 INTEGER 列（无需 numeric 迁移、无截断坑），且短视频本来就按秒说（15s/30s/60s）。prompt 侧无需改（它只吃 `targetWordCount`，不吃分钟）。

**改动**（动了 shared+jobs → 需 Trigger 重部署；含 DB 迁移+回填 ×60）：`schema/poet.ts`（两表列改名）、`schemas/poet.ts`（`computeTargetWordCount` 除 60、`min(15).max(3600)`）、`generate-script.ts`、tRPC `schemas/poet.ts`+`routers.ts`、`write-script-button.tsx`+`custom-topic-actions.tsx`（DURATIONS 改秒 + 自定义输入）、若干展示页。

**顺带（归大改但槽已在）**：`poet_custom_topics.durationMinutes/targetWordCount` 在生产流程里是**死字段**（只采集于写稿时）。"时长前置到分析阶段"只需让 analyze 写入这两列、写稿时读回作默认 —— 正好佐证客户"逻辑倒置"的判断，且 schema 槽现成。

## #7 SOP 对齐 —— 实证：SOP 确实被用上

**代码 trace（确凿）**：写稿时 `generate-script.ts:196-201` 查 `clerk_sops`（channelId + `sopType=ai_reference`，取最新），经 `scriptWriter` 传入，并在三个 prompt 里真实插值（`poet.ts:102` 短稿 / `:173` 长稿大纲 / `:233` 分段）。两条路径（Muse 选题 / 自定义选题）都走这段，行为一致。

**生产实证**：23 个频道 **21 个有 ai_reference SOP**（仅相机口播 / AI效率 两个 xhs 没有）；近 8 篇脚本 **7 篇带 sopId**、1 篇空。→ **SOP 实际是被用上的，客户"被忽略"的怀疑基本不成立。**

**真问题**：频道若**没有** ai_reference SOP，写稿会**静默用空字符串继续**（`generate-script.ts:202` 只 warn 不报错），脚本照样产出、看起来"成功" —— 这才是该补的。次要：自定义选题写稿时忽略 `topicRow.sopId`（总用最新）、SOP 查询无语言过滤（旧的中文 ai_reference 可能漏进来）。

**推荐**：① UI 在写稿前提示"该频道还没有 AI 参考 SOP，建议先用 Clerk 生成"（把静默变显性）；②（可选）写稿前置校验。无需改写稿核心逻辑。

**改动**：前端提示（纯前端）+ 可选后端校验。
