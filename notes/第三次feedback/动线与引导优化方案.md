# Singularity 动线与引导优化方案

> 范围：全中文 UI 的小型创作者工具，三层 IA（Clerk 全局引擎 / 我的账号 资产层 / Project 执行单元）已落地。本方案不引入 tour 库，不推倒重来，全部为渐进式改造现有组件 + 空状态内嵌按钮 + 完成态 CTA + 导航重组 + 上下文头部 + 文案统一。所有落点均为已核实的真实文件路径。

---

## 1. 现状动线一页图

理想黄金路径（owner 认可）：

```
进入工作台 → 建自有账号 → 建项目 → 项目内绑对标 → Clerk 拆解出 SOP
            → 给账号生成圣经 → Muse 出选题 → Poet 写稿
```

当前实际动线与断点（✗ = 用户在此卡住 / 茫然 / 走错路）：

```
[OAuth 回调]
  └─ /welcome  1500ms 纯动画 splash，无一字说明产品做什么          ✗ gap-4

[首次进入工作台 / channelCount===0]
  └─ 只有「Singularity 围绕你自己的频道运转」一句 + 一个建频道按钮  ✗ gap-5
     无三 agent 角色、无四步流程示意；侧边栏 5 个裸标签看不懂      ✗ #1 sidebar-labels
     侧边栏分组「全局/我的账号」，账号被压到第二组末项            ✗ #2 痛点2
     移动端导航整条藏在无标签汉堡键后                              ✗ gap-7

[建账号]  /accounts/new 标题「新建频道」按钮「创建频道」           ✗ channel-vs-account 命名矛盾
  └─ 成功 → router.push('/accounts') 朴素列表 + 一闪 toast         ✗ #3 痛点3
     /accounts 空状态零文字说明「账号是什么、为何先建」            ✗ accounts-list-empty

[项目 Hub]  /accounts/[slug]/projects/[project]
  └─ Muse / Poet 两卡等权平铺，无「先做哪个」顺序                  ✗ #6 setup-checklist
     Clerk 入口在本页完全缺席；对标卡沉在最底                      ✗ #6
     绑完对标点完成，无「下一步去 Muse」提示                       ✗ bind-competitors-to-muse

[Clerk]  /clerk 列表只含自有账号；用户以为能「拆解对标」           ✗ gap-1 概念根因
  └─ /sops 空态按钮「去分析对标」其实 Clerk 不分析对标（文案撒谎）  ✗ gap-2
  └─ /clerk/[slug] 站内最长页之一，无分节导航，SOP 沉底且默认折叠  ✗ clerk-section-nav
     分析完成只弹纯文字 toast，全页无指向圣经/项目的链接          ✗ #4 痛点4

[圣经]  /accounts/[slug]/bible
  └─ activeBible 为 null 时空态纯文字，生成按钮在右上角 header     ✗ sop-bible-empty-cta
     生成/激活后无「去 Muse 出选题」桥                            ✗ bible-to-muse-bridge

[Muse]  /accounts/[slug]/projects/[project]/muse
  └─ 未绑对标时空态：纯文字「先在项目里绑定对标账号」无按钮        ✗ #5 痛点5（最高优先）
     审过选题后无「去 Poet 写稿」桥                               ✗ muse-to-poet-bridge
     巡视完成 toast 无下一步 CTA                                  ✗ long-task-completion

[Poet]  /accounts/[slug]/projects/[project]/poet
  └─ 全空兜底「先生成圣经 → 通过 Muse 选题 → …」全是纯文字          ✗ #6 poet-empty-states
     无圣经时「新建自定义选题」禁用且无就地生成入口                ✗ poet-empty-states
     无 SOP 写稿用原生 confirm，且无去 Clerk 入口                  ✗ no-sop-native-confirm
     写稿完成只 router.refresh，无「查看脚本」CTA                  ✗ long-task-completion

[贯穿全程]
  └─ 长任务跨页后失联，全局 shell 无运行指示、无找回路径          ✗ gap-10/11/12
     「对标」四种叫法、「圣经/Bible」中英混用、返回按钮三种范式    ✗ 一致性组
```

核心结论：现有 issue 大多是「做完一步缺下一步按钮」的接驳，但 **gap-1/2 揭示了一个概念根因——用户以为 Clerk 拆解的是「对标」，代码里 Clerk 永远只分析自有账号**。这不是缺按钮，是 Clerk/对标/Muse 三者数据关系与用户心智不一致。本方案在 P0 先用文案统一口径止血。

## 2. 设计原则（5 条，从 issues 归纳）

1. **每个空状态必须可行动**：不允许「纯文字告诉用户去某处自己找路」。空状态要么内嵌跳转按钮，要么内嵌就地触发器（如 BibleGenerateSheet）。死胡同一律配 CTA。
2. **每个完成态必须给下一步**：长任务完成的瞬间是引导黄金窗口。toast 用 sonner 的 `action` 升级为带跳转；页面在产出区末尾放静态「下一步」卡。
3. **口径全站唯一**：同一实体只有一个对外名（账号 / 对标账号 / 圣经），返回按钮只有一种范式（返回 + 目的地名）。文案不能与能力背离（Clerk 不分析对标就不能写「分析对标」）。
4. **导航顺序即黄金路径**：从上到下、从首屏到次屏的视觉权重，要让「此刻该做的事」高于「并列工具入口」。
5. **引导克制、不打扰**：用内联文案、空状态按钮、静态卡、可选 toast action；禁用强制 tour、弹窗轰炸、营销标语/badge。引导文字只说清「这是什么、下一步去哪」。

### P0 — 黄金路径断点与 owner 点名痛点（高影响、低复杂度优先）

- **落点**：`muse/page.tsx`、`muse/_components/muse-run-button.tsx`
- **复杂度**：S

#### P0-2 · 建完账号被甩回死列表 → 直跳项目 Hub + 方向 toast（痛点3）
- **问题**：新建账号成功 `router.push('/accounts')` 落到朴素列表，只有一闪 toast。
- **改法**：`create-channel-form.tsx` 第 39-43 行 `onSuccess`，跳转目标改为新账号默认项目 Hub（spine 保证默认项目 slug == 账号 slug，已核实），slug 做 `encodeURIComponent`。

     - ① 绑定对标（boundCount>0 ✓，href `#competitors`）
     - ② 用 Clerk 拆解频道生成 SOP（clerkSopCount>0 ✓，href `/clerk/[a]`）— **措辞为「拆解频道」不写「分析对标」**（Clerk 作用于自有频道，见 gap-1）

  4. `<ProjectCompetitorsCard>` 外层包 `<div id="competitors">` 供步骤① 锚点。
  5. 可选：Muse/Poet 卡前置条件未满足时降为 `opacity-60`（次要）。
- **文案**：标题「上手清单」；各步如上，不加 badge/标语。
- **落点**：`projects/[project]/page.tsx`、新建 `setup-checklist.tsx`、`accounts/_components/project-competitors-card.tsx`、`packages/db/src/schema/{project,competitor}.ts`

### P1 — 导航 / IA / 布局层级

#### P1-1 · 侧边栏重组（合并工作项：分组顺序 + 标签角色化 + 命名统一）
合并 sidebar-grouping-order / sidebar-labels-tooltips / sidebar-missing-muse-poet / competitor-terminology（侧边栏部分），单文件 `components/app-sidebar.tsx` 一次改完。

- **分组顺序**（痛点2）：账号上提至工作台正下方，导航从上到下即黄金路径开头。
- **标签角色化/去黑话**（痛点1）：「Clerk」裸英文 + 「SOP 库/对标账号池」黑话 → 角色化；统一「·」中点。
- **不做 tooltip**：`<Sidebar>` 默认 `collapsible="offcanvas"`，桌面端无 collapsed 图标栏态，tooltip 永不显示（已核实）。如需 hover 解释，用 `<Link>` 原生 `title` 属性（可选，优先级低于标签）。
- **不加 Muse/Poet 侧边栏项**：`/muse`、`/poet` 是 `permanentRedirect`，且 bare 路由不按账号数深链——直接进侧边栏会让单账号用户被丢到账号列表，比现状更差。若一定要补 agent 对等入口，**必须先修 bare 重定向**（复用 page.tsx solo 深链逻辑：恰好 1 个账号时深链到 `/accounts/{slug}/projects/{slug}/muse`），属可选增强，不在本轮必做。

最终分组/顺序 ASCII：

```
Singularity
┌─ 开始 ──────────────────┐
│  ⌂  工作台                │  /
│  📺 我的账号              │  /accounts   ← 上提，紧贴工作台
├─ 分析与素材 ────────────┤
│  🔍 Clerk · 分析师        │  /clerk

│  🎯 对标账号              │  /competitors  ← 「对标账号池」→「对标账号」

```

- **改法**：将 ACCOUNTS 的「账号」并入第一组第二位（删独立 ACCOUNTS 组），label 改「我的账号」；第一组 `SidebarGroupLabel`「全局」→「开始」，第二组「分析与素材」=[Clerk · 分析师, SOP 库, 对标账号]；「账号」item 的 `isActive` 保持前缀匹配（不加 `end:true`，否则进 `/accounts/[slug]` 丢高亮）。
- **落点**：`apps/web/components/app-sidebar.tsx`
- **复杂度**：S

#### P1-4 · Clerk 分析页分节导航 + SOP 提权（clerk-page-section-nav-sop-buried）

## 4. 明确不做（防范围蔓延）

1. **不引入 tour 库**（driver.js / intro.js 等）。所有引导用现有 shadcn/base-ui 组件、空状态内嵌按钮、下一步卡、toast action、导航重组、文案。

3. **不做侧边栏 Muse/Poet 项**（除非先修 bare `/muse`、`/poet` 重定向的 solo 深链）。本轮二者入口靠工作台卡片 + 项目下钻 + ContextHeader 面包屑承载。
4. **不做侧边栏 tooltip**：offcanvas 模式下永不显示，无效投入。
5. **不做 Muse `#competitors` hash 锚点跳转**：ProjectCompetitorsCard 异步加载，hash 滚动落点不准（项目 Hub 内部清单的锚点因同页、组件已挂载，可做）。
6. **不物理上移 Clerk SOP section**：换位致两平台 IA 不一致，用锚点 + 摘要徽章 + 默认展开解决「埋」。

8. **不改 `idea-approve-toggle` 的 onSuccess 跳转**：保留 router.refresh，审选题时不应被拽离页面。
9. **不重写 NextStepCard 的 pickStep 单步逻辑**：单步选择是对的（避免打扰），只在其上叠加全局进度。
10. **不改 `/welcome` 为产品介绍页**：它是 OAuth 过场，方向感前移到工作台 0 账号空状态。
