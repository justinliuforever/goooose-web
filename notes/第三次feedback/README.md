# 第三次反馈 · 处理说明

## A. 反馈全文梳理（确认无遗漏）

### 核心痛点（3）

1. **「频道（Channel）」是多义词** —— 在 Clerk（对标账号）/ Muse / 工作台（自有账号）多处出现，每次含义不同，造成认知歧义。
   - 归属：整体框架大改（重命名 + 分层）。

2. **底层逻辑倒置** —— 视频时长应在「分析选题」时确定，现在却拖到最后「写稿」才限制。
   - 归属：拆两半 —— ≤60s / 自定义时长属重构前优化；真正「前置到分析阶段」归整体大改。

3. **SOP / Bible 无法全局沉淀、跨项目复用** —— Clerk 的 SOP 与 Poet 的 Bible 缺乏全局沉淀和跨项目引用能力。
   - 归属：整体框架大改（SOP 解耦 / Project 引用）。

### 体验优化

**Clerk（账号 / 内容分析）**

- ① **术语翻译腔**：社会仪式 CTA / 签名式动作 / 主题聚类 / 认知基模炸弹 等自造词过于生硬，要换行业通用说法（直接 CTA / IP 标志性动作 / 常见主题·核心话题 / 黄金前 3 秒钩子·完播率痛点·避免观众跳出）。
  - 方案：升级中文 wrapper（术语表 + 去翻译腔约束），重跑生成后生效。根因与 A/B 验证见验证版 #1。
- ② **「主题」功能定位**：它是归纳已分析视频的共同点，并非给账号做大类目分类。
  - 方案：改「主题聚类」措辞 + 补一句定位说明文案。见验证版 #2。
- ③ **性能**：实测 20 视频 / 1 小时，需优化。
  - 已查清 root cause（见 [性能调查-clerk.md](./性能调查-clerk.md)）：**小红书路径完全串行、没有并发**（YouTube 才有 pLimit(4)），叠加每条 Pro@8192 分析 ~2 分钟、XHS 每条白跑一次 Deepgram。首选修复=XHS 并行化（48 分钟 → ~10–12 分钟）。**已确认实施方案见该文 §4 表**（与重构正交，可先行实施）。

**Muse（选题灵感）**

- **入口与操作迷茫**：用户在 Muse 看到的是 Clerk 分析过的账号，但找不到「导入对标账号」的入口；去频道页也只有「新建频道」没有「导入」。
  - 方案：加「导入对标账号」入口。**归 Part 2（整体大改）** —— 随重构一次性建 `competitor_accounts` 一等表，不做临时版，Part 1 不实施。见验证版 #3。

**Poet（写稿）**

- ① **Bible 多版本并行**：想用同一爆款框架（如「红发魔女」）写不同领域（相机 / AI 产品）的选题，现在只能覆盖或重新生成；且生成速度偏慢。
  - 方案：加「+ 新建 Bible」+ 历史版本列表（DB 已支持）；生成改用 Flash + streaming 提速。见验证版 #4。
- ② **Bible 激活前可查阅 / 编辑**：现在只有激活后才能看细节和编辑。
  - 方案：未激活版本也允许查阅 / 编辑（后端本就不限，纯前端）。见验证版 #5。
- ③ **时长前置 + ≤60s 自定义**：长短视频的分析与生成逻辑本质不同；现在最短预设 5 分钟，短视频普遍 60s 以内，必须支持更短的自定义跨度。
  - 方案：改 `durationSeconds`（整数秒）支持 ≤60s + 自定义；真正「前置到分析阶段」归整体大改。见验证版 #6。
- ④ **术语对齐**：写稿是否真的调用了 Clerk 输出的 SOP 规范，需要交叉测试。
  - 已验证：SOP 确实被写稿用上（21/23 频道有 SOP、7/8 脚本带 sopId）；真问题是无 SOP 时静默用空串。见验证版 #7。

## C. 整体框架大改 —— 采用方案一（已定方向）

**采用客户推荐的方案一**，三层架构：

1. **Clerk = 全局分析引擎**：不属于任何账号 / 项目，产出可被任意项目引用的 **SOP 库**。
2. **我的账号 = 资产层**：「频道」正名「我的账号」，沉淀**多版本 Bible**。
3. **Project = 执行单元**：在某个「我的账号」下新建，绑定 SOP + Bible + 对标账号 + 平台 / 目标时长；**Muse、Poet 是 Project 内的并列工具**。

详细数据模型 / 页面层级 / 迁移路径 + 我方优化点（A–F + 3 决策）见 [重构设计-方案一.md](./重构设计-方案一.md)。设计已确认，实施阶段再细化 schema/线框、再动代码。
