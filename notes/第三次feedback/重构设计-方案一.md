## 1. 三层架构

1. **Clerk = 全局分析引擎**：不属于任何账号 / 项目，产出可被任意 Project 引用的 **SOP 库**。
2. **我的账号 = 资产层**：「频道」正名，沉淀**多版本 Bible**。
3. **Project = 执行单元**：属于某个「我的账号」，绑定 SOP + Bible + 对标账号 + 平台 + 目标时长；**Muse、Poet 是 Project 内的并列工具**。

## 2. 数据模型

| 实体 | 说明 | 关系 |
| --- | --- | --- |
| **对标账号** competitor_accounts | 一等实体，Clerk 分析 + Muse 监控 + Project 引用**共用一张表**（B） | 属于 user；可被多 Project 引用 |
| **Clerk 分析 + SOP** | SOP 归属其来源（通常是对标账号；ad-hoc URL 分析 → "未命名源"）（A, Q1） | SOP →(归属)→ 对标账号；SOP ↔ Project **多对多引用** |
| **我的账号** own_accounts | 创作者自有账号，沉淀多版本 Bible | 属于 user；含多 Project |
| **Bible** | 多版本，账号级资产 | Bible →(归属)→ 我的账号；Project 选用其中一份 |
| **Project** | 一个平台 + 目标时长（默认，可覆盖）（D, Q2） | 属于一个我的账号；引用 SOP(多)/Bible(一)/对标账号(多) |
| Muse 选题 / Poet 脚本 | 在 Project 内产出 | →(归属)→ Project |

关系草图：
```
user
├─ 对标账号[]            ←── Clerk 分析 → SOP[]（SOP 溯源到对标账号）
└─ 我的账号[]
   ├─ Bible[]（多版本）
   └─ Project[]（一平台 + 目标时长）
      ├─ 引用: SOP[]（多对多）/ Bible(一) / 对标账号[]
      ├─ Muse 选题[]
      └─ Poet 脚本[]
```

## 3. 设计决策（A–F + Q1–Q3）

- **A｜SOP 可溯源**：SOP 归属来源对标账号（可增量重分析），不是无主；"库"= 跨账号可浏览可选的池子。ad-hoc URL 分析归到"未命名源"。
- **B｜对标账号唯一一等实体**：Clerk / Muse / Project 共用一张 `competitor_accounts`，不建两套。
- **C｜nav 分层**：Clerk 与对标账号池**全局置顶**；Muse / Poet 在 Project 内。比"三个 agent 全改 project"更准。
- **D｜时长前置**：Project 级"平台 + 目标时长"被 Muse / Poet 继承 → 天然解决痛点 2，无需等写稿。
- **Q1｜SOP 引用**：SOP ↔ Project **多对多**（一 Project 多 SOP、一 SOP 多 Project）。
- **Q2｜Project 边界**：**一 Project = 一平台**；时长 = **项目级默认 + 单条脚本可覆盖**。
- **E/Q3｜迁移**：见 §5。

## 4. 页面 / 导航

左侧：
```
全局
├─ Clerk（分析引擎）
└─ 对标账号池
我的账号
└─ <账号> → Project[] → 进入 Project → [Muse | Poet]
```
- Clerk 出 SOP 进库；Muse 在 Project 内选 SOP + 导入对标账号 → 选题；Poet 在 Project 内绑 SOP + Bible → 写稿。

## 5. 迁移（enterprise：expand-migrate-contract，零丢失、可回滚）

1. **expand**：新表（own_accounts / competitor_accounts / projects + 引用 join）与现有 `channels` 等**并存，不删**。
2. **migrate（回填）**：
   - 每个现有 `channel` → 1 个我的账号 + 1 个**默认 Project**（包住该账号现有的 SOP 引用 / Bible / 选题 / 脚本，避免孤儿）（E）。
   - `channels.competitors` JSONB 每条 → 1 个对标账号 + link。
   - 现有 `clerk_sops` → 关联到对应对标账号；建 Project↔SOP 引用。
3. **兼容期**：旧 `channelId` 读路径与新模型并行，UI 逐步切换。
4. **contract**：全部验证通过后再删旧列 / 旧表。
